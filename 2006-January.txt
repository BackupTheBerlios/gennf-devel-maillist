Hallo Otto,

Das Problem mit putenv ist zum einen, dass der uebergebene String in
der Umgebung landet, es also ein und GC stehendes Lisp Objekt sein darf,
weil dann Umgebungsvariablen verschwinden. Du kannst ja mal in den
osicat-patch Code hineinschauen, ich habe jetzt einfach dort ein malloc
platziert (was auch durch ein uffi:allocate-foreign-string geschehen
koennte). Wenn jetzt die Umgebungsvariable umdefiniert wird, ist mir
nicht klar, ob der alte String delokiert wird. Ich wuesste auch nicht,
wie ich das machen soll ausser mit einem kruden

char *p = getenv(name);
free(p-sizeof(char)*strlen(name)-1);

was auch nur funktionieren wurde, wenn die Umgebungsvariable von unserem
Programm selber angelegt wurde.

Weiterhin funktioniert das entfernen von Umgebungsvariablen nicht, da
Solaris putenv die Variable nicht entfernt, wenn putenv(name) aufgerufen
wird. Auch hier tritt gleichzeitig dasselbe Problem mit der
Speicherplatzfreigabe auf wie oben.

Irgendwie ist das etwas bescheuert. Fuer uns ist es letztlich nicht so
wichtig, da wir es vermeiden koennen, Umgebungsvariablen zu setzen. Ich
haette das nur gerne ordentlich repariert (d. h. nur putenv benutzen, da
es das offensichtlich auf allen Systemen gibt), damit man nachher ein
vernuenftiges Patch fuer die Bibliothek hat.

Bis nachher,

Florian

Hallo,

habe gerade die Dokumentation zu distributed repositories gelesen. Hoert
sich gut an, ein Feature, das bisher scheinbar noch nicht angesprochen
wurde (und soweit ich es eben getestet habe, noch nicht in mcvs
implementiert ist), scheint mir ein moeglicher subdirectory-checkout zu
sein.

Also, bei gegebender directory-struktur im modul fnord:
documentation/barf.tex
documentation/foobar.txt
foo.txt
source/bar.lisp
source/foobar.lisp

faende ich es toll, wenn man durch
mcvs -d:ext:somewhere:/mcvs co fnord/documentation
nur das documentation directory auschecken koennte.

Ich denke, das man das dadurch loesen kann, die metadata der files (MAP
file, etc) in jedem Verzeichnis zu speichern, anstatt in einem zentralen
MCVS Repository.

Im MCVS directory von fnord muesste man dann nur Informationen ueber
foo.txt und die directories documentation und source halten.

Die Changes muesste man dann natuerlich auch dezentral speichern,
genauso wie die rcs-files.

Der primaere Vorteil waere sicherlich, dass man bei einem beliebig
grossen Repository nicht immer das komplette auschecken muss, sondern
auch Teilbaeume auschecken kann.

Haette zusaetzlich den Vorteil, dass man dann auch auf directory-Ebene
branches erzeugen koennte. Wobei ich denke, wir sollten die branches
schon Repository-weit machen (wobei man natuerlich auch branches von
subdirectories machen will, also, nicht jeder branch muss das gesamte
repository umfassen).

Aber ich denke, auch dieses kann eine weitere Ausbaustufe sein,
moeglicherweise koennte man dann die branches wie bei subversion (durch
ein Verzeichnis branches) realisieren, aber der im scratchpad
beschriebene Vorschlag, branches zu implementieren gefaellt mir deutlich
besser.

Die Changeset-Nummer muss natuerlich immer repository-global vergeben
werden (und eindeutig in einem repository sein).


Bis morgen,

hannes

Hannes Anmerkung ist richtig, dass Sandboxes, die nur Unterverzeichnisse
enthalten, nicht beruecksichtigt sind. Meta-CVS kann das von Haus aus:

$ mcvs -d repo co project subdir/subsubdir

erzeugt ein Verezeichnis subsubdir, mit den entsprechenden Dateien.
Allerdings ist das CVS sandbox im MCVS-Unterverzeichnis nach wie vor
komplett. Das entspricht dann nicht ganz dem, was Hannes meint.

Ich glaube aber, dass Auschecken von Unterverzeichnissen sich einfach
dadurch aufbauen laesst, dass eben nur der entsprechende Teil der
F-Files ausgecheckt wird, der einen entsprechenden Pfad hat.
Ich wuerde naemlich ungerne Unterverzeichnisse im Repository haben, auch
nicht fuer Branches. Ich kann das nicht genau begruenden, habe nur das
Gefuehl, dass wir mit nur einer Indirektionsstufe besser fahren.
In einem teilweise ausgechecktem Repository muss natuerlich bekannt
sein, dass nicht alle Dateien da sind, das entspricht in etwa der
DISPLACED-Datei, die Meta-CVS fuer diese Zwecke einsetzt (allerdings nur
zur Manipulation von Pfaden).

Zum Thema Branches auf Unterverzeichnisse wurde ich sagen, dass man das
durch eine Art Lazy-Branching erledigen koennte: Ein Branch wird ueber
das gesamte Repository angelegt, es werden aber nur fuer die Files
tatsaechlich F-Files angelegt, die veraendert werden, fuer alle anderen
wird dies verzoegert und bei Bedarf nachgeholt (mglw. bringt uns das
allerdings die Links wieder hinein, die eigentlich nicht benoetigt sind
-- sozusagen als eine Art QUOTE). Auch hier denke ich, dass wir
konzeptionell nicht so eingeengt sind, das wir das nachher nicht mehr
unter bekommen.

Hallo Hannes, hallo Otto,

ich habe die die Befuerchtung, dass wir ein echtes Problem haben, was
damit zusammenhaengt, wie wir Changes abspeichern. Dabei wird eine
Abbildung Dateiname->Revisionsnummer gebildet und abgespeichert. Welche
Revisionsnummer eine Datei bei einem Commit bekommt, wissen wir aber
immer erst hinterher. Das bedeutet, dass der Change erst nach dem Commit
gebildet und im Repository abgelegt werden kann. Scheitert das aus
irgendeinem Grund, liegt ein nicht registrierter Change im Repository.
Den koennte man wieder ruckgaengig machen, sozusagen eine Art
registrierter Rollback. Wenn aber der Commit des Change vorher nicht
klappte, ist es gut moeglich, dass der Rollback auch nicht klappt.

Eine andere Moeglichkeit ist, zuerst einen trockenen Commit zu machen
und sich nur die Revisionsnummern zu holen und dann alle Files und den
Change in einem Atemzug einzuchecken. Es kann aber sein, dass jemand
dazwischen kommt und dann die Revisionsnummern nicht mehr stimmen. Das
koennte man nach dem Commit ueberpruefen und so zumindest den Fehler
feststellen. Ich weiss allerdings gerade nicht, wie man dann damit
umgehen kann.

Diese Umstaende einfach zu ignorieren, scheint mir wirklich keine gute
Loesung zu sein.
habt Ihr Ideen, wie man aus dem Dilemma herauskommt?
Es ist zwar so, dass CVS sowieso keine atomaren Commits kann aber wir
sollten nicht noch so einen Problemklotz oben draufsetzen.

Dann noch etwas anderes: Ich hatte am Dienstag die Vermutung geaeussert,
dass es moeglicherweise besser ist, nocht direkt an den Meta-CVS Code
anzubauen, sondern nur die fuer uns brauchbaren Teile daraus zu
uebernehmen (entweder als Ideen oder direkt als Code).
Dafuer sprechen folgende Punkte, die mir am Code nicht gefallen:
* CVS ist fest einverdrahtet.
* Pfadnamen werden als Strings behandelt. Das Umgehen mit Pfaden ist
  eines der Kernpunkte dieses Programms, deswegen finde ich es besonders
  ungluecklich, dass hier nicht mit der Pfadnamen Abstraktion gearbeitet
  wird.
* Die Unix-Schnittstelle ist unguenstig (die ist zwar nun portiert, aber
  nur so, dass es mit SBCL laeuft, nicht so, dass es eine gute Loesung
  ist).
Das sind alles mehr oder weniger Geschmacksfragen und betreffen im
Wesentlichen das Design. Es gibt aber noch einen weiteren Punkt, der
nicht nur mit Design zu tun hat:
Meta-CVS ueberlaesst es im Wesentlichen CVS, zu ueberpruefen, welche
Dateien eingecheckt werden muessen und welche nicht. Dabei wird dafuer
gesorgt, dass das Mapping zwischen F-Files und Dateien in der Sandbox
konsistent bleibt.
Wir schieben an dieser Stelle eine Ebene dazwischen, naemlich
die oben angesprochene Abbildung (die das Locking Problem verursacht).
Wir wissen also immer genau, welche Aenderungen gemacht wurden und
koennen explizit sagen, welche Dateien eingecheckt werden muessen,
welche aktualisiert werden muessen, geloescht wurden, hinzukamen,
verschoben wurden usw. usf. Und das muessen wir auch so machen, weil
sonst alles durcheinander kommt. Das bedeutet auch, dass Verschieben,
Loeschen, Hinzufuegen von Dateien in einer Sandbox nur ueber von uns
bereitgestellte Kommandos geschen darf, damit das entsprechend
registriert werden kann. Der ganze Code, um die Sandbox mit dem Mapping
abzugleichen und umgekehrt, ist also ueberfluessig.
Die enge Anbindung an CVS scheint mir hier eher hinderlich.

Mein Vorschlag ist deswegen, die Idee von Meta-CVS zu uebernehmen, alle
Dateien als F-Files abzulegen, eine MAP einzufuehren und die Dateien mir
Hardlinks in die Sandbox einzuklinken. Hier koennen wir auch Code
uebernehmen, denke ich. Der ganze Rest, die Anbindung an das Backend,
das Handling von Branches und Merges (diesbzgl. bringt uns Meta-CVS
sowieso nichts) wird neu gemacht.

Um nocheinmal auf das eingangs erwaehnte Problem zu kommen: Das kriegen
wir natuerlich auch nicht damit in den Griff, dass wir mehr
neuschreiben, sondern nur dadurch dass wir irgendwie ein Locking
einbauen. Das geht aber meiner Ansicht nur, indem wir CVS nicht mehr so
benutzen, wie das ueblicherweise gemacht wird. Wir muessen so eine Art
P, V Operationen haben und das Repository abschliessen, bevor wir
anfangen, daran Modifikationen zu machen. Und irgendwie muss
sichergestellt werden, dass die V-Operation immer ausgefuehrt wird und
das Repository in jedem Fall in einem konsistenten Zustand ist.

Moeglicherweise sieht das gerade alles ziemlich konfus aus, aber ich
habe massive Zweifel, dass das so geht, wie wir uns das vorgstellt haben
(und das liegt nicht nur daran, dass ich zu faul bin 5000 Zeilen
Meta-CVS Code zu lesen und zu kapieren -- ich bin mir naemlich nicht
sicher, ob uns das ueberhaupt weiterbringt).

Meldet Euch,

Florian

~ * globaler lock (fuer das repo bzw das subdir/den branch)

Hallo Otto,

der erste Branch bekommt die Nummer 1. Wenn ein neues Repository
angelegt wird, wird auch gleich dieser Branch 1 erzeugt. Es sind dabei
aber zwei Operationen im Spiel:

1) Repository anlegen. Die Operation gibt es auch schon. Sie legt das
entsprechende cvs Modul an und legt eine leere BRANCH Datei ab sowie ein
ACCESS file, dass den cvs root String fuer dieses Modul enthaelt.
2) Einen leeren Branch in dem Repository anlegen. Wenn das Repository
vorher {} war, so enthaelt es danach eben den Branch 1, der aber noch
leer ist. Es wird also ein Verzeichnis 1 angelegt sowie ein noch leeres
CHANGE File. An dieser Operation arbeite ich gerade.

So etwas wie ein cvs import beinhaltet dann die Operationen 1 und 2
sowie ein nachfolgendes hinzufuegen aller Dateien, die importiert werden
sollen. In den ersten Branch. Hinzufuegen und andere Operationen, die
irgendwie mit Dateien der Benuzter hantieren, gibt es alle noch nicht.

Ich hoffe, dass beantwortet die Frage.

Florian
