Message-ID: <200602121830.k1CIUVLi024502@sheep.berlios.de>

florenz     2006/02/12 19:30:30 CET

  Modified files:
    .                    prototype/backend-cvs.lisp 
                         prototype/gennf.lisp 
  Log:
  I have found a solution for the checkout/commit problem.
  The trouble was that co/up -r sets sticky tags so that committing
  with these files is not possible.
  As cvs is no longer responsible for sensible or insane commits
  a routine that checks out an older change in a way that committing
  with these files is possible is necessary.
  The first approach was to use -p option to co/up and write into
  appropriate files. But this omits entries in cvs' meta data and
  committing with these files is not possible.
  What is done now is perhaps not very beautiful but seems to work:
  If no revision is given, co/up -jHEAD is executed which yields
  the same result as co/up without any option. If a certain revision
  is required (because a certain change is required)
  co/up -jHEAD -j1.n is passed to cvs. As the checkout-change function
  is general and defaults to latest change if no told otherwise, two
  -j options are passed to cvs in any case.
  An exception is for the change, access and branch files because
  they are not included in a change and it is always the
  latest revision which is fetched.
  Furthermore, there are two backend-cvs routines for checkout: one uses
  cvs co, the other one uses cvs up. This is because for the first
  checkout co has to be used to build the cvs meta durectory etc.
  For subsequent calls up has to be used to keep those meta data intact.
  The routine cvs-get automatically dispatches to the correct routine
  (depending on the existence of a CVS directory in the destination
  directory).

  Log:
  We have now a two-way merge algorithm implemented.
  Therefore, we have file merging.lisp which performs a two-way merge
  either on files or sequences.
  In particular, the following happens:
  (two-way-merge file1 file2) means that file1 is merged into file2.
  A diff of the two files is computed using cl-difflib (an implementation
  of the Gestalt Pattern Matching algorithm by Ratcliff and Obershelp).
  All chunks of file1 that can be inserted into file2 are inserted.
  All chunks of file1 that conflict with a chunk in file2 are
  insrted annotated.
  All chunks of file2 that do not exist in file1 are inserted
  with an annotation.
  The annotations are produced by functions which can be passed to
  two-way-merge.
  Furthermore, it is possible to tell two-way-merge, if it shall
  produce a merge despite conflicts or not. If the corresponding argument
  is nil, a merge is only produced if no conflicts arise.
  The return value of two-way-merge is a multiple-value with the
  first being the merge and the second indicating if conflicts arose or not.
  
  I am not sure, if I broke some parts of the code today, may be it is
  inconsistent after this commit.
  
  Next step is probably three-way-merge. This is due to the fact, that on
  sandbox update or on commits, there is in fact an ancestor which can
  be used during merge. This can in fact be handled by cvs but I think it is
  better that all mergeing is either done by cvs or by gennf. And for
  branching/merging it cannot be done by cvs, so it should be done by gennf
  for update and commit as well.
